tried spacebot with discord and used it for a day. its far from ready (unreliable, bad sec architecture, poor usablity, many things that dont actually work) but  it's a first  usable POC giving an idea what this could feel like. A more coherent repackaging of the claw-osphere and  with a UI. 
some observations:
- i hate non deterministic things that don't have to be. we have workflow engines and UI for visibilty,  there is no excuse to hook an agent to cron and try it make the system goo brrr when its just slow, expensive, unreliable and does not bring any advantage
- virtual, ever lasting conversations that compact in the background and pretend to be continuous are weird. there is definitely room for it but not everywhere and not as transparent as most attempts try to make it. this might change but not in the near term. power users need to see compaction markers, constext size and overview/some control whats in it.
- tiny subagents for all the small tasks just rocks. its one of the things that made amp great
- bigger sub-worker agents for the main task out of band of the main conversation is great but only if you can switch to the subagent any time and the parent agent can also see what your interactions there were 
- working with agents in discord group channels as teams or community  is really interesting, there are tons of possibilities that could emerge, we have to try this
- cli first tools just don't work, either they can't do anything or they are insecure, sandboxing does not help because you need to give them data. this cli hype will die for sure
- memory systems without version control are unimaginable now. databases can just be disposable indices but not the main data. life cycles on memory systems need to be integrated in the version control workflows similar to software develeopment pr/ci/cd.
- i think most agents should not get any context up front except the agent system prompt at all.  it just depends too much on the situation what is relevant and it feels so much cleaner to always know you start with the same tiny and consistent state.  models are so good at gathering the context they need after the first user message that its nearly always better to have the bit higher latency.  (especially with pre-contexting subagents that have a list of common cases and can search and summarize out of band)  there can also be guardrails that force skill inclusion like pulling in a svelte skill if svelte is mentioned or .svelte files are used.
