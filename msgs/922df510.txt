Looking for comments/ feedback on the security architecture for darc draft:
- all internal/pre IWA communication happens via unix sockets with filedescriptors in the app home dir including dev tool protocol
- on installation, the darc daemon sets up up a client server self signed cert for mTLS with a browser policy to use for internal connections, this sets up a trusted connection of daemon to the IWA context, preventing network sniffing etc
- on first connection users will not have a device identity and prompted with webauthn to create passkey > did:key
- the daemon sets up the did:key as owner for all resources it controls, eg its docker context, all browser reseource, data
- user can optionally link the key to their atproto identity, link up other devices for recovery or 2nd factors
- every website resource / permission request or agent permission is a UCAN delegation/invocation from the owner did
- UCANs can be delegated across devices and also to servers, if user adds a home-server that runs an agent they can delegate a permission to the that agent 
- this has plenty advantages, even things that are just nice side effects eg. if you allow a scheduled action, it would not be able to run outside of the allowed time window even if the schedule misfires
- root and high level UCANs have  UI  modelled after credit card purse in apple wallet
- delegations and invocations have UI modelled after event/movie ticket cards in apple wallet and some of the UI concepts apple developed for kerberos and mozilla for webid (foaf + ssl) but obviously much simplified as both had major usabilty issues for users
